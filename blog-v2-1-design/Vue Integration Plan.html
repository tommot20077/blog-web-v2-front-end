§ 02 — config/site.ts

把所有開關集中到一個 config。

single source of truth

你提到色系想在 config 裡設定。我擴大這個概念：所有「站台層級」的開關都應該收到 src/config/site.ts。包含主題預設值、accent 色、feature flags、SEO meta、社群連結、analytics ID。這樣換站、開新站、A/B 測試都從一個檔案動手。

export type AccentKey = 'blue' | 'green' | 'red' | 'purple' | 'amber'
export type SiteConfig = typeof SITE

composable 從 config 讀預設值，使用者選擇覆蓋寫到 localStorage：

src/composables/useTheme.ts 
import { ref, watchEffect } from 'vue'
import { SITE, type AccentKey } from '@/config/site'

type Mode = 'light' | 'dark' | 'sepia'

// 從 config + localStorage 讀初始值
const stored = (k: string, fb: string) =>
  typeof localStorage !== 'undefined'
    ? localStorage.getItem(k) ?? fb
    : fb

const mode   = ref<Mode>(stored(SITE.theme.persistKey, SITE.theme.defaultMode) as Mode)
const accent = ref<AccentKey>(stored('sp.accent', SITE.theme.defaultAccent) as AccentKey)

watchEffect(() => {
  document.documentElement.setAttribute('data-theme', mode.value)
  document.documentElement.setAttribute('data-accent', accent.value)
  localStorage.setItem(SITE.theme.persistKey, mode.value)
  localStorage.setItem('sp.accent', accent.value)
})

export function useTheme() {
  return {
    mode, accent,
    available: SITE.accents,
    canChange: SITE.theme.allowUserOverride,
    cycleMode: () => {
      const order: Mode[] = SITE.features.sepiaMode
        ? ['light', 'dark', 'sepia']
        : ['light', 'dark']
      mode.value = order[(order.indexOf(mode.value) + 1) % order.length]
    },
    setAccent: (a: AccentKey) => { accent.value = a },
  }
}
§ 為什麼這樣分 config = 站長決定的，code-time 設定（要不要 sepia、預設色、品牌名）。localStorage = 讀者個人偏好的 runtime 覆蓋。Feature flag 在 config 關掉時，整個對應的 UI（讀者切換器、admin 的設定面板）都會消失，不需要動別的程式。
§ 03 — Routes & Layouts

4 個批次對應的路由表。

vue-router

把 prototype 的所有頁面映射成 Vue Router 的 routes，並用 nested layout 把 public / admin 分開。

src/router/index.ts 
import { createRouter, createWebHistory } from 'vue-router'
import { SITE } from '@/config/site'

export default createRouter({
  history: createWebHistory(),
  routes: [
    // ─── Public layout ───────────────────────────
    {
      path: '/',
      component: () => import('@/layouts/PublicLayout.vue'),
      children: [
        { path: '',           name: 'home',    component: () => import('@/views/HomeView.vue') },
        { path: 'posts/:slug', name: 'post',    component: () => import('@/views/PostView.vue') },
        { path: 'about',      name: 'about',   component: () => import('@/views/AboutView.vue') },

        // Batch 1 — site pages
        { path: 'archive',    name: 'archive', component: () => import('@/views/ArchiveView.vue') },
        { path: 'tags/:tag?', name: 'tags',    component: () => import('@/views/TagsView.vue') },
        { path: 'search',     name: 'search',  component: () => import('@/views/SearchView.vue') },
        { path: 'privacy',    name: 'privacy', component: () => import('@/views/PrivacyView.vue') },

        // Batch 2 — reader features (gated by config)
        ...(SITE.features.readingList ? [{
          path: 'reading-list', name: 'reading-list',
          component: () => import('@/views/ReadingListView.vue'),
        }] : []),
      ],
    },

    // ─── Admin layout ────────────────────────────
    {
      path: '/admin',
      component: () => import('@/layouts/AdminLayout.vue'),
      meta: { requiresAuth: true },
      children: [
        { path: '',           redirect: '/admin/dashboard' },
        { path: 'dashboard',  component: () => import('@/views/admin/DashboardView.vue') },
        { path: 'drafts',     component: () => import('@/views/admin/DraftsView.vue') },
        { path: 'scheduled',  component: () => import('@/views/admin/ScheduledView.vue') },
        { path: 'media',      component: () => import('@/views/admin/MediaView.vue') },
        ...(SITE.features.newsletter ? [{
          path: 'newsletter', component: () => import('@/views/admin/NewsletterView.vue'),
        }] : []),
        { path: 'authors',    component: () => import('@/views/admin/AuthorsView.vue') },
      ],
    },

    // ─── 404 fallback ───
    { path: '/:pathMatch(.*)*', component: () => import('@/views/NotFoundView.vue') },
  ],
})

§ 04 — Prototype 對應表

每一個 prototype 檔案的去處。

where it lands
Prototype（React）Vue 落點備註
enhancements.css → src/styles/enhancements.css 純 CSS，直接搬。最先做這個。
enhancements.jsx (ToastProvider) → useToast.ts + ToastStack.vue 已在 migration kit 提供。
enhancements.jsx (ConfirmProvider) → useConfirm.ts + ConfirmModal.vue 跟 toast 同模式，全域單例。
enhancements.jsx (MobileDrawer) → MobileDrawer.vue Teleport 到 body，scroll-lock 用 onMounted。
v3/command-palette.jsx → CommandPalette.vue 用 Vue 的 defineModel 做 v-model:open。
v3/article-extras.jsx (useAutoTOC) → useAutoTOC.ts + StickyTOC.vue onMounted 掃 DOM，用 IntersectionObserver。
v3/article-extras.jsx (CodeBlock) → CodeBlock.vue 建議搭配 Shiki 做 build-time highlight。
v3/ai-seo.jsx (SEO) → @unhead/vue 的 useHead Vue 生態系標準工具，比 React Helmet 好用。
v3/ai-seo.jsx (MarkdownView) → routes: /posts/:slug.md Server route 直接回 Markdown content-type。
site-pages/batch1.jsx → views/Archive · Tags · Search · NotFound · Privacy 5 個 view，HTML/CSS class 直接搬，把 useState 換成 ref。
site-pages/batch2.jsx → views/ReadingListView + 一堆 composables highlight、last-read、print、theme picker 拆 5 個檔。
site-pages/batch3.jsx → views/admin/* + components/admin/* 整批拆成 5 個 view + 共用元件，吃 AdminLayout。
data.jsx (假資料) → src/data/posts.ts 或 API 之後接真實 CMS 時改這裡，view 不動。
§ React → Vue 轉換心法 1. JSX → template：className 改 class、{ } 改 {{ }}、onClick 改 @click。
2. useState → ref：const [x, setX] = useState(0) 變 const x = ref(0)，setX(v) 變 x.value = v。
3. useEffect → watchEffect：不需要 deps array，自動追蹤。
4. Provider → provide/inject 或全域單例：單例最簡單，prototype 用很多 Context 都可以這樣。
§ 05 — Migration Plan

兩週完成，每天 2–3 小時。

14 days

把整合工作拆成 6 個階段。每階段獨立可上線，做完一個就 deploy 一次，永遠不會卡住。

§ 01 · Day 1

Tokens + Config

建 src/config/site.ts。把 enhancements.css 整包搬進來，補 sepia + 5 個 accent variants。useTheme 改吃 config。

§ 02 · Day 2–3

Composables + 全域元件

useToast、useConfirm、useReadingProgress、useHotkeys。對應的 ToastStack、ConfirmModal、ReadingProgress、CommandPalette 全部接到 PublicLayout。

交付：⌘K 可叫出，toast 可彈

Batch 1 — 站台基本盤

Archive、Tags、Search、Privacy、404 五個 view。Search 用 FlexSearch 或 Pagefind。404 用 router catch-all。

交付：站長能找到任何文章
§ 04 · Day 6–8

Batch 2 — 閱讀產品化

useReadingList、useHighlights、useLastRead 三個 composable + 對應 UI。print stylesheet。i18n 用 vue-i18n 起骨架。

交付：讀者有「自己的閱讀帳號」
§ 05 · Day 9–11

Batch 3 — 後台

建 AdminLayout（含 router auth guard）。Dashboard、Drafts、Scheduled、Media、Newsletter、Authors 六個 view。資料先 mock，之後接 API。

§ 06 · Day 12–14

SEO + AI + Polish

@unhead/vue 接 SEO meta。/llms.txt、/feed.xml、/sitemap.xml 三個 server route。OG image generator。a11y 過一遍。

交付：SEO 滿分，AI 友善，可上線
§ 開始前的兩個決定 1. 你目前 Vue 專案是純 SPA 還是 Nuxt？如果是 Nuxt，SEO meta 用 useSeoMeta、server routes 直接寫在 server/api/。如果是純 Vite + vue-router，需要外掛 @unhead/vue 跟自己處理 sitemap.xml 的 build step。
2. 後台的 auth 怎麼接？prototype 假設已登入。實際上 AdminLayout 需要一個 router guard 檢查 token。先用 mock（localStorage 一個布林值）開發，最後一階段再接真實 OAuth。
