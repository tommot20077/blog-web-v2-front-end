關於把 prototype 注入到 repo 的折衷美學

當設計團隊跟工程團隊各自迭代到一個段落，會出現一種「我們的不是同一個東西」的錯覺。但其實只要 token 對齊、字體對齊，剩下的差異都只是版面選擇 — 是可以慢慢收斂的。

出發點：兩條平行線

設計團隊用 prototype 工具迭代視覺與互動，工程團隊用 Vue 跟 Vite 寫真實的 view。一段時間之後，雙方手上的「同一個產品」會開始長得不一樣 — 不是因為誰錯了，而是因為兩邊的工具在解決不同問題。

這個 sample 想證明：融合不需要重做。把 prototype 當作 idea source，把 repo 當作 truth source，就能維持工程的整潔，又同時把設計的好點子塞進去。

折衷美學的可能性

「折衷」聽起來像是一個負面詞 — 但實際上，好的設計幾乎都是折衷的結果。我們不會無視 token、不會無視既有元件、不會無視已經寫好的 routing — 我們在這些限制裡面，找出最舒服、最一致的下一步。

以這份 ArticleDetail 為例，repo 已經有了 hero、cover、prose、reading progress bar 跟右側 navigator dot。prototype 想加的東西其實只有四個：左側 TOC、右側 ActionBar、選文字的 highlight toolbar、以及 last-read 指示。

注入點 1 — TOC

左側 TOC 用 IntersectionObserver 抓 h2/h3 的進入狀態，當前段落會把左邊的 indicator border 染成 accent 色。沒有伺服端、沒有 manual config，純靜態頁也能用。

注入點 2 — ActionBar

右側的 action bar 取代 repo 原本的 dot navigator（在桌面尺寸下兩者擇一）。Bookmark / Like / Share / View as Markdown 都是讀者每天會用的動作，把它們從散落的位置集中到一個 vertical rail。

注入點 3 — Highlight & note

選一段文字後會浮出一個 toolbar，可以選顏色、加註記、複製。註記儲存在 localStorage 裡（之後也可以同步到 server）。這對長文閱讀來說特別實用 — 想引用某句話、想標記一段重點，不用切回 Notion。

注入點 4 — Last-read indicator

離開時記得讀者捲到哪一段（用 IntersectionObserver 抓最後可見的 heading），下次回來時用一個小 toast 提示「上次讀到這」，並在文章中對應位置畫一條細線標記。

為什麼這四個是「對」的注入點

因為它們都符合三個條件：

  • 不破壞現有結構：prose 的 markup 不變、hero 的 layout 不變
  • 用 design token，不另開色票：highlight 的三色、toolbar 的 ink/bg 都是現成的
  • 能漸進式啟用：使用者可以在 Settings → Appearance 開關
「最好的注入是讀者察覺不到的注入。當他開始用了才發現：這個東西原本好像就在這。」

下一步

把這四個 component 變成獨立的 Vue SFC：ArticleTOC.vue / ArticleActionBar.vue / HighlightLayer.vue / LastReadIndicator.vue。每一個都接受一個 prop（文章 id 或 ref），透過 composable 讀寫 localStorage。整合到既有的 ArticleDetailView.vue 只需要四行 import。

之後可以再決定要不要把這些資料同步到 server — 但那是另外一個 milestone 的事了。
